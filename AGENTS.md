# Repository Guidelines

## Ground Rules
- **Source of Truth**: Treat `README.md` as the startup source of truth. If this file diverges, follow `README.md` and update this file.
- **Tech Stack**: Use **uv** (package manager), **polars** (dataframes), **PostgreSQL** (runtime pipeline db), and **Alembic** (schema migrations).
- **Hard Constraints**: 
  - Normalize inputs before writing to the database.
  - Enforce a policy that new columns, tables cannot be added without explicit approval.  EXPLICITLY STATED!!
  - Never apply raw `ALTER TABLE` manually — generate a migration with `alembic revision --autogenerate -m "description"` and apply with `alembic upgrade head`. (Migration scripts live in `alembic/versions/`. Review autogenerated migrations before committing).
  - Do not reintroduce SQLite logic.
  - Never fail silently. Log all issues, including failed inserts/upserts and zero-result/zero-row operations, only use `loguru`.
  - Keep secrets in `.env`; never log credentials; respect site rate limits; run scrapers in headed mode if blocked.
  - Keep implementation lean for this one-engineer project: **no schema churn, no heavy framework**; prefer practical PostgreSQL checks integrated into existing `controller`, `recovery`, and `db_audit` paths with tests.

## Pipeline Success Criteria
A successful pipeline run is measured by **data completeness**, not by steps completing without errors. The chain of title and encumbrance data are the core deliverables. Without them, the pipeline produces no investment-grade analysis. Judgment PDFs and enrichment data are intermediate steps toward that goal.

**After any pipeline run, you MUST validate thresholds:**
- Use `db_audit.py` for validation. Only `db_audit.py` can determine success.
- If the run fails the audit:
  1. Diagnose why the data is missing (check loguru logs, find root cause).
  2. Fix the root cause.
  3. Keep iterating until thresholds are met.

## Canonical Commands

- **Full pipeline (Bulk Refresh + Enrichment):** `uv run Controller.py`
- **Force all loaders:** `uv run Controller.py --force-all`
- **Quick sanity run:** `uv run Controller.py --auction-limit 5 --judgment-limit 5 --ori-limit 5 --survival-limit 5 --limit 5`
- **Run Scheduled Jobs Manually:** `uv run python -m src.tools.run_scheduled_job --job <job_name>`
- **Run Web App / Dashboard:** `uv run python -m app.web.main`
- **Run Database Migrations:** `uv run python -m src.db.migrations.create_foreclosures --dsn <postgres-dsn>`
- **Linting & Typing:** `uv run ruff check .` (add `--fix` when safe); `uv run ty check` for typing; `uv run pytest` for unit tests.

## Multi-LLM Workflow Rules
- AGENTS MUST ALWAYS ASK TO TAKE REPRESENTATIVE SAMPLES if they plan on doing so.
- Do NOT rely on internal, hidden markdown artifacts (such as `walkthrough.md`, `task.md`, or scratchpads) for architectural documentation. ALL PLANS and DOCUMENTATION must be in the repository
- If you design a new system, fix a complex bug, or discover important project context, you MUST write that documentation directly into the repository `docs/` folder and link it in `README.md`.
- Other LLMs are reading this codebase, so all critical knowledge must be surfaced in the repository itself.
- Python source files that implement architecture or complex behavior MUST include a clear, detailed module-level docstring at the top of the file explaining architectural purpose and how the file fits into the broader system.

## Testing & Commit Guidelines
- Use pytest; name files `test_*.py` mirroring module paths.
- Prefer schema/column presence checks and aggregate assertions for pipelines; include fixture HTML/PDF snippets for scrapers in `tests/fixtures/`.
- Run `uv run pytest` plus a quick `Controller.py` sanity run before PRs.
- After every change, run `uv run ruff check .` and `uv run ty check`.
- Commits: short, present-tense verbs (e.g., `Add lien summary parser`); keep scope focused.

## Web-Readiness Checklist
Before claiming web completion, verify these are true in code and UI:
1. Property detail uses real `property_market` payload (not placeholder-only snapshot shaping).
2. Encumbrances render from `ori_encumbrances` with survival statuses.
3. `property.sources` provenance is populated when source data exists.
4. NOCs are populated in permits tab.
5. Tax tab lien rows are populated.
6. Chain rows provide document links when files are available.
